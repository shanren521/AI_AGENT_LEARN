# 团队知识库智能问答（RAG + Agent）完整技术要点

这是现在 AI Agent / AI 应用开发岗位里非常高频的项目方向。

很多公司所谓：

* AI Copilot
* 企业知识助手
* 内部 GPT
* 智能文档助手
* 企业 AI 搜索

本质上都是：

```text
RAG + Agent + Workflow
```

---

# 一、系统本质

本质是：

```text
用户问题
   ↓
Query Understanding
   ↓
Retrieval（RAG）
   ↓
Tool / Agent Planning
   ↓
LLM Synthesis
   ↓
Answer + Citation
```

---

# 二、核心架构

```text
                +------------------+

         +-------------------+
```

# 难点

## PDF

最恶心。

因为：

```text
PDF ≠ 文本
```

问题：

* 双栏
* 表格
* 页眉页脚
* OCR
* 图片
* 编码

---

## 面试高频

### Q：PDF解析为什么难？

回答：

```text
PDF 本质是排版格式，不是语义格式。
需要解决 layout、table、OCR、reading order 等问题。
```

# 2. Chunking（切块）

这是 RAG 最核心。

很多人只会：

```python
RecursiveCharacterTextSplitter
```

但生产远远不够。

---

# 为什么 Chunk 很重要

因为：

```text
Embedding 只能表达局部语义。
```

chunk 不合理：

* 召回错
* 上下文断裂
* hallucination

---

# Chunking 策略

---

## 1. Fixed Chunk

```text
每500 token切一块
```

简单。

缺点：

```text
语义断裂
```

---

## 2. Semantic Chunk

按：

* 标题
* 段落
* 语义

切分。

生产更常见。

---

## 3. Hierarchical Chunk

这是大厂常见。

结构：

```text
Document
  ├── Section
  │      ├── Paragraph
  │      ├── Paragraph
```

检索：

* 先粗召回
* 再细召回

---

# 面试高频

## Q：为什么 chunk overlap 有必要？

因为：

```text
信息可能跨 chunk 边界。
```

Overlap 能减少语义截断。

# 5. Retrieval（真正核心）

大量 RAG 项目失败：

不是模型问题。

而是：

```text
召回失败
```

---

# 常见方案

---

## 1. Dense Retrieval

Embedding 相似度。

问题：

```text
关键词精确匹配差
```

---

## 2. Sparse Retrieval

BM25。

问题：

```text
语义能力差
```

---

## 3. Hybrid Retrieval（生产推荐）

```text
BM25 + Vector
```

这是企业最常见。

# 6. Rerank（极其重要）

这是很多初级开发不会的。

---

# 为什么需要 rerank

向量检索：

```text
召回 ≠ 最相关
```

必须：

```text
二次排序
```

---

## 常见：

* bge-reranker
* Cohere rerank
* cross encoder

---

# 面试高频

## Q：为什么需要 rerank？

因为：

```text
embedding retrieval 是粗召回。
cross-encoder 才是真正相关性排序。
```

---

# 7. Agent 部分

这是现在和传统 RAG 最大区别。

---

# 传统 RAG

```text
Query → Retrieve → Answer
```

---

# Agentic RAG

```text
Query
  ↓
Planner
  ↓
Multi-Step Retrieval
  ↓
Tool Calling
  ↓
Reasoning
  ↓
Answer
```

---

# Agent 会做什么

---

## Query Rewrite

用户：

```text
“那个报销流程”
```

Agent：

```text
“2025 差旅报销审批流程”
```

---

## Multi-hop Retrieval

例如：

```text
A文档引用B文档
```

需要多步推理。

---

## Tool Calling

Agent 可能：

* 查 Jira
* 查 GitLab
* 查数据库
* 查 API

---

# 面试高频

## Q：Agent 和传统 RAG 区别？

传统 RAG：

```text
单次检索
```

Agentic RAG：

```text
规划、多步推理、工具调用
```

# 8. 权限系统（企业最重要）

很多 demo 根本没考虑。

生产必须：

---

## 文档权限隔离

例如：

```text
HR文档
财务文档
研发文档
```

不能互相看到。

---

# 常见做法

---

## Metadata Filtering

```python
{
  "department": "engineering"
}
```

检索时过滤。

---

# 面试高频

## Q：如何防止越权检索？

回答：

```text
retrieval 阶段做 metadata filter，
而不是生成后再过滤。
```

这是重点。

# 9. Hallucination（幻觉）

企业最怕：

```text
AI胡说
```

---

# 常见方案

---

## 1. 强制引用

```text
Answer with citations
```

---

## 2. Context Constrained Generation

```text
只能基于提供上下文回答
```

---

## 3. Confidence Score

低置信度：

```text
“不确定”
```

---

# 面试高频

## Q：如何降低 hallucination？

标准答案：

* 高质量 retrieval
* rerank
* citation
* constrained generation
* answer verification

# 六、面试最容易被问的问题

---

# RAG 基础

---

## Q1 RAG 为什么比 Fine-tune 更适合知识库？

因为：

* 实时更新
* 成本低
* 不需要重新训练
* 可追溯

---

## Q2 Chunk 大小怎么选？

看：

* embedding model
* 文档类型
* retrieval 场景

通常：

```text
300~800 tokens
```

---

## Q3 为什么 Hybrid Search 很重要？

因为：

```text
Dense retrieval 不擅长精确关键词。
```

---

## Q4 为什么 rerank 很关键？

因为：

```text
第一阶段 retrieval 是高召回，
rerank 才是高精度。
```

---

# Agent 部分

---

## Q5 Agentic RAG 为什么比普通 RAG 强？

因为：

* query rewrite
* decomposition
* multi-hop
* tool use

---

## Q6 Agent 为什么容易失控？

因为：

```text
tool loop
无限推理
prompt injection
```

---

## Q7 如何限制 Agent？

必须：

* max iterations
* timeout
* tool whitelist
* sandbox

---

---

核心：

```text
分层索引
混合检索
异步embedding
增量更新
```

---

## Q9 如何提升召回率？

常见：

* metadata filtering
